## 1. Задача: предсказание следующего слова

### 1.0 Откуда взялась идея: Марков и «Евгений Онегин»

В 1913 году русский математик **Андрей Марков** взял поэму Пушкина «Евгений Онегин» и вручную подсчитал, как часто одна буква следует за другой. Гласная за гласной, согласная за гласной — он составил таблицу переходов.

Это была первая **языковая модель**: вероятность следующей буквы зависит от текущей.

```
Цепь Маркова 1-го порядка:
  P(буква_{t+1} | буква_t)

Пример: P('а' | 'к') = 0.31   ← после 'к' чаще всего идёт 'а'
        P('о' | 'к') = 0.24
        P('и' | 'к') = 0.18
        ...
```

Марковская цепь порождает текст так:
1. Стоим в букве «к»
2. Выбираем следующую случайно, по вероятностям из таблицы
3. Переходим в неё и повторяем

**Проблема:** цепь помнит только 1 предыдущее слово. Рассмотрим два предложения:

```
«Нападающий забил в финале»     → после «забил» ждём «гол», «мяч»...
«Плотник забил в стену»         → после «забил» ждём «гвоздь», «дюбель»...
```

Цепь Маркова не различает эти контексты: для неё оба раза просто «после слова забил». Без предыдущего слова («нападающий» vs «плотник») предсказание случайное.

**Что изменилось за 110 лет:**
- Марков: контекст = 1 буква
- n-gram модели (1990-е): контекст = 3–5 слов
- RNN (2010-е): контекст = несколько предложений, но плохая память
- **Трансформер (2017–сейчас): контекст = тысячи токенов, точно**

### 1.1 Математическое доказательство Маркова

Чтобы понять, почему цепь Маркова нетривиальна, воспроизведём оригинальный расчёт 1913 года. Марков превратил текст «Евгения Онегина» в математический объект: последовательность из $N = 20\,000$ символов, каждый из которых принимает одно из двух состояний — гласная ($v$) или согласная ($c$).

**Эмпирические частоты:**

$$n_v = 8\,638 \quad \text{(гласных)}, \qquad n_c = 11\,362 \quad \text{(согласных)}$$

Эмпирическая вероятность встретить гласную и согласную:

$$P(v) = \frac{n_v}{N} = \frac{8\,638}{20\,000} \approx 0.43, \qquad P(c) = 1 - P(v) \approx 0.57$$

**Гипотеза о независимости.** Оппоненты Маркова утверждали: появление букв — независимые события. Тогда вероятность пары «гласная + гласная» должна равняться произведению:

$$H_0: \quad P(v, v) = P(v) \cdot P(v) = (0.43)^2 \approx 0.19$$

В тексте из $N = 20\,000$ символов содержится $N - 1 = 19\,999$ соседних пар. **Ожидаемое** число пар «гласная + гласная» при независимости:

$$E(v, v) = 19\,999 \cdot 0.19 \approx 3\,731$$

**Реальный подсчёт.** Марков вручную подсчитал фактическое число таких пар:

$$O(v, v) = 1\,104$$

Сравнение:

$$O(v, v) \ll E(v, v) \qquad (1\,104 \neq 3\,731)$$

Фактическое количество пар гласных оказалось **более чем в три раза меньше** ожидаемого — разница, которую нельзя списать на случайность. **Вывод: буквы не являются независимыми событиями.** В тексте наблюдается явная тенденция к чередованию гласных и согласных.

**Марковская цепь.** Раз буквы зависимы, вероятность следующего символа определяется текущим. Это и есть определение цепи Маркова 1-го порядка:

$$P(X_{n+1} \mid X_n, X_{n-1}, \dots, X_1) = P(X_{n+1} \mid X_n)$$

Система описывается уже не одной вероятностью, а **матрицей переходов**:

$$P = \begin{pmatrix} P(v \mid v) & P(c \mid v) \\ P(v \mid c) & P(c \mid c) \end{pmatrix}$$

Из-за тенденции к чередованию диагональные элементы (та же буква за той же) малы, а перекрёстные — велики:

$$P(v \mid v) < P(c \mid v), \qquad P(c \mid c) < P(v \mid c)$$

Именно этот простой расчёт — $1\,104 \neq 3\,731$ — позволил Маркову строго доказать, что закон больших чисел применим к **зависимым** событиям, навсегда изменив теорию вероятностей.

---

Вся мощь современных LLM (Large Language Models) сводится к одной удивительно простой задаче:

> **Дан текст. Предсказать следующий токен.**

Это называется **авторегрессионное языковое моделирование**. Пример:

```
Вход:  «Кошка сидит на»
Выход: «коврике» (или «диване», «столе» — модель выдаёт вероятности)
```

Во время генерации текста это повторяется в цикле:

```
«Кошка»          → «сидит»
«Кошка сидит»    → «на»
«Кошка сидит на» → «коврике»
...
```

Каждый шаг — один forward pass через нейросеть. Это и есть то, что вы видите при «печатании» ChatGPT — он буквально выдаёт по одному слову (вернее, токену) за раз.

**Почему это работает?**

Задача кажется ограниченной, но на самом деле она охватывает всё:
- Чтобы правильно закончить предложение на юридическую тему — нужно знать право
- Чтобы написать код — нужно знать языки программирования
- Чтобы ответить на вопрос — нужно хранить знания о мире

Всё это «выучивается» из гигантских корпусов текста только за счёт предсказания следующего слова.

## 2. Классификация и градиентный спуск

Прежде чем разбирать Word2Vec и трансформер, нужно понять одну механику, которая лежит в основе **всего** современного глубокого обучения: как параметры модели обновляются через градиентный спуск. Разберём на примере многоклассовой классификации — именно такая задача стоит и в Word2Vec, и в любом LLM.

---

### 2.1 Задача: определение жанра текста

Дан фрагмент текста. Нужно предсказать жанр — один из четырёх:

```
Класс 0: наука
Класс 1: спорт
Класс 2: политика
Класс 3: технологии
```

Признаки текста — вектор чисел: `x = [x₁, x₂, x₃]`. Например:
- `x₁` = частота слов «эксперимент», «данные», «исследование»
- `x₂` = частота слов «гол», «матч», «команда»
- `x₃` = частота слов «президент», «выборы», «закон»

**Модель** — линейный слой, который для каждого класса вычисляет отдельный **score (логит)**:

$$z_k = w_k \cdot x + b_k \quad \text{для каждого класса } k \in \{0, 1, 2, 3\}$$

`w_k` — вектор весов для класса `k`, `b_k` — его смещение. Это параметры модели.

В матричном виде: `z = W · x + b`, где `W` — матрица размером `(4 × 3)`.

Scores — просто произвольные числа (могут быть отрицательными). Чтобы превратить их в вероятности, применяем **Softmax**.

---

### 2.2 Softmax: из scores в вероятности

$$P(k \mid x) = \frac{e^{z_k}}{\sum_{j=0}^{3} e^{z_j}}$$

Softmax гарантирует:
- Все вероятности от 0 до 1
- Сумма по всем классам = 1
- Класс с наибольшим score получает наибольшую вероятность

**Пример.** Модель выдала scores: `z = [2.1, 0.5, -0.3, 1.8]`

```
exp(z):    [8.17,  1.65,  0.74,  6.05]
сумма exp: 16.61

P(наука)       = 8.17 / 16.61 = 0.492  ← самый высокий
P(спорт)       = 1.65 / 16.61 = 0.099
P(политика)    = 0.74 / 16.61 = 0.045
P(технологии)  = 6.05 / 16.61 = 0.364
```

Модель думает: 49% — наука, 36% — технологии. Разумно.

---

### 2.3 Функция потерь: Cross-Entropy

Допустим, правильный ответ — **класс 0 (наука)**. Нам нужно число, которое измеряет «насколько плохо» предсказание:

$$\text{Loss} = -\log P(\text{правильный класс} \mid x) = -\log P(0 \mid x)$$

В нашем примере:

```
Loss = -log(0.492) ≈ 0.71    ← неплохо, правильный класс получил почти 50%
```

Другие примеры:

| P(правильного класса) | Loss | Интерпретация |
|-----------------------|------|---------------|
| 0.95 | 0.05 | Модель уверена и права — отлично |
| 0.50 | 0.69 | Неуверенна, но склоняется к правильному |
| 0.10 | 2.30 | Почти уверена в неправильном — плохо |
| 0.01 | 4.61 | Катастрофа |

Логарифм штрафует нелинейно: чем увереннее модель в неправильном ответе, тем больше штраф.

---

### 2.4 Градиентный спуск: как обновляются веса

После вычисления Loss нужно понять: **в какую сторону** изменить каждый вес, чтобы Loss уменьшился?

Для этого вычисляем **градиент** — производную Loss по каждому параметру:

$$\frac{\partial \text{Loss}}{\partial W_{ki}}$$

Это число говорит: «если увеличить этот вес на ε, Loss изменится на `ε × gradient`».

**Правило обновления (SGD):**

$$W \leftarrow W - \eta \cdot \frac{\partial \text{Loss}}{\partial W}$$

где `η` — learning rate, например `0.01`.

**Интуиция:** поверхность функции потерь — это многомерный «ландшафт». Градиент указывает направление подъёма. Мы идём в противоположную сторону — вниз.

## 3. Word2Vec: откуда берутся векторы слов

Прежде чем перейти к трансформеру, разберём Word2Vec (Mikolov et al., 2013) — модель, которая показала: если нейросеть учится предсказывать контекст слова, она **сама** строит геометрически осмысленные векторы. Именно эта идея лежит в сердце всех современных LLM.


---

### 3.1 Задача: предугадай соседей

Возьмём предложение:

```
«Кошка    быстро    бежала    по    комнате»
   0         1         2       3       4
```

Алгоритм **Skip-gram** (самый распространённый вариант Word2Vec):

> Дано центральное слово → предскажи слова в окне ±2 вокруг него.

Например, для центрального слова «бежала» (позиция 2), окно ±2 даёт:

```
бежала  →  должен предсказать: «Кошка», «быстро», «по», «комнате»
```

Это порождает обучающие пары `(центральное, контекстное)`:

```
(бежала, Кошка)
(бежала, быстро)
(бежала, по)
(бежала, комнате)
(быстро, Кошка)
(быстро, бежала)
(быстро, по)
...
```

Миллиарды таких пар извлекаются из текстового корпуса автоматически, без разметки.

---

### 3.2 Архитектура: две матрицы

Word2Vec — это нейросеть с одним скрытым слоем. Но слой особый.

**Параметры модели** — две матрицы:

```
W_in:  (vocab_size × dim)    ←  «входные» эмбеддинги
W_out: (vocab_size × dim)    ←  «выходные» эмбеддинги
```

Для конкретности: `vocab_size = 10 000`, `dim = 300` (типичные значения):

```
W_in:  (10000 × 300)  →  3 000 000 параметров
W_out: (10000 × 300)  →  3 000 000 параметров
```

**Обе матрицы инициализируются случайными числами** — маленькими, около нуля.

---

### 3.3 Forward pass: одно предсказание шаг за шагом

Берём обучающую пару: центральное = «бежала» (индекс 4821), контекстное = «быстро» (индекс 712).

**Шаг 1. One-hot вектор центрального слова**

```
x = [0, 0, ..., 1, ..., 0]   ← 10 000 нулей и одна единица на позиции 4821
```

Это «вопрос»: какое слово мы обрабатываем?

**Шаг 2. Умножение на W_in → скрытый вектор h**

```
h = x · W_in    →    shape: (1, 10000) × (10000, 300) = (1, 300)
```

Поскольку x — one-hot, это умножение **эквивалентно просто выбору строки 4821** из матрицы W_in.

```
h = W_in[4821, :]    ← просто 300 чисел, строка «бежала»
```

Именно поэтому W_in и есть таблица эмбеддингов: каждая строка — вектор одного слова.

**Шаг 3. Умножение на W_out → оценки (scores)**

```
scores = h · W_out^T    →    (1, 300) × (300, 10000) = (1, 10000)
```

Это скалярное произведение вектора «бежала» с каждым столбцом W_out.  
Результат: вектор из 10 000 чисел — оценка того, насколько каждое слово является хорошим соседом.

**Шаг 4. Softmax → вероятности**

```
P(слово_j | бежала) = exp(scores_j) / Σ_k exp(scores_k)
```

Получаем вектор из 10 000 вероятностей (каждая от 0 до 1, сумма = 1).

**Шаг 5. Вероятность правильного ответа**

Нас интересует `P(«быстро» | «бежала»)` = `P(j=712 | «бежала»)`.

В начале обучения (случайные веса) эта вероятность ≈ 1/10000 = 0.0001 — почти ноль.  
**Цель обучения:** сделать её как можно ближе к 1.

---

### 3.4 Функция потерь: Cross-Entropy Loss

Нам нужно число, которое показывает «насколько плохо» предсказание.

**Cross-Entropy** для одного примера:

```
Loss = -log P(правильный_ответ)
```

**Почему логарифм?**

| P(правильный) | -log(P) | Смысл |
|---------------|---------|-------|
| 0.0001 | 9.21 | Совсем неправильно, большой штраф |
| 0.01 | 4.61 | Плохо |
| 0.5 | 0.69 | Средне |
| 0.99 | 0.01 | Почти правильно, малый штраф |
| 1.0 | 0.0 | Идеально |

Логарифм «штрафует» за уверенность в неправильном ответе нелинейно: при P=0.0001 штраф в 9000 раз больше, чем при P=0.9.

**Пример вычисления:**

```
scores(«бежала») = [-0.3, 0.1, ..., 0.8(«быстро»=712), ..., -0.1]

exp(scores):      [0.74, 1.11, ..., 2.23, ..., 0.90]
сумма exp = 8750  (по всем 10 000 словам)

P(«быстро» | «бежала») = 2.23 / 8750 = 0.000255

Loss = -log(0.000255) = 8.27    ← большой — обучение только началось
```

---

### 3.5 Обратное распространение: как обновляются матрицы

После вычисления Loss, вычисляем **градиент** — насколько нужно изменить каждое число в W_in и W_out, чтобы Loss уменьшился.

**Идея градиентного спуска:**

```
θ ← θ - lr · ∂Loss/∂θ
```

Для нашей пары «бежала» → «быстро»:

- Вектор «быстро» в W_out: **увеличиваем** (он должен быть «ближе» к «бежала»)
- Векторы всех других слов в W_out: **уменьшаем** чуть-чуть (они «слишком близко» к «бежала»)
- Строка «бежала» в W_in: обновляется так, чтобы следующий раз scores(«быстро») был выше

После нескольких миллионов примеров:
- Строки «бежала», «прыгала», «несла» в W_in начинают быть похожими (у них общие контексты)
- Строки «быстро», «медленно», «осторожно» группируются рядом (они встречаются в похожих контекстах)

---

### 3.6 Почему вектора становятся осмысленными

Центральная идея: **слова, которые встречаются в похожих контекстах, получают похожие векторы**.

```
«бежала»   встречается рядом с: {быстро, медленно, кошка, собака, ...}
«прыгала»  встречается рядом с: {быстро, высоко, кошка, мяч, ...}
«летела»   встречается рядом с: {быстро, высоко, птица, самолёт, ...}
```

«бежала» и «прыгала» имеют много общих контекстных слов → их векторы в W_in сближаются.

Это известно как **дистрибутивная гипотеза** (Harris, 1954):
> «Слово узнаётся по своей компании» — слова с похожим значением встречаются в похожих контекстах.

Word2Vec — это просто алгоритм, который реализует эту гипотезу через матричные операции.

---

### 3.7 Знаменитый пример: король − мужчина + женщина = королева

После обучения на большом корпусе можно делать **арифметику над смыслами**:

```
вектор(«король») - вектор(«мужчина») + вектор(«женщина») ≈ вектор(«королева»)
```

**Как это читать:**

```
вектор(«король»)  содержит: [власть, мужской_пол, монархия, ...]
вектор(«мужчина») содержит: [мужской_пол, ...]
разность:                   [власть, монархия, ...]   ← «королевскость без пола»
+ вектор(«женщина»):        [власть, монархия, женский_пол]
```

Эта сумма геометрически близка к вектору «королева», потому что «королева» встречается в тех же контекстах, что «власть + монархия + женщина».

**Другие примеры арифметики:**
```
Париж - Франция + Германия ≈ Берлин      (столицы)
программист - мужчина + женщина ≈ программистка
кошка - котёнок + щенок ≈ собака          (детёныш → взрослое)
```

Никто специально не объяснял модели что такое «страна» или «столица». Это **возникло само** из задачи предсказания контекста.

---

### 3.8 Negative Sampling: особенность Word2Vec

В отличие от современных LLM, которые на каждом шаге спокойно считают softmax по всему словарю (матмул на GPU — дёшево), Word2Vec обучается через **Negative Sampling**. Это связано с эпохой: 2013 год, CPU, словарь 1–3 миллиона слов — полный softmax был реально медленным.

Идея:
1. Берём одно **правильное** контекстное слово (позитивный пример)
2. Случайно сэмплируем K=5–20 **неправильных** слов (негативные примеры)
3. Учим модель давать правильному слову высокий score, а негативным — низкий

```
Loss = -log σ(score(«быстро»))           ← правильное: хотим score → +∞
      -Σ log σ(-score(«холодильник»))    ← неправильные: хотим score → -∞
      -Σ log σ(-score(«синтаксис»))
      ...
```

По сути — бинарная классификация «это правильный контекст или нет», а не предсказание по всему словарю. Та же идея позднее легла в основу contrastive learning (CLIP, SimCLR).

---



### 3.9 Итог: что такое эмбеддинг

**Эмбеддинг** — это строка матрицы W_in для данного слова после обучения. Это просто список чисел (300 или 2048), который:
- Инициализировался случайно
- Обновлялся миллионы раз через gradient descent
- В итоге «вобрал» статистику контекстов слова из корпуса

Вектор не несёт никакого «заранее заложенного смысла». Смысл — это геометрическое расположение относительно других векторов, которое возникло из данных.

---

### 3.10 Разница с эмбеддингами в LLaMA

В Word2Vec:
- Только одна задача: предсказать контекстное слово
- Только одна матрица W_in (используется после обучения как эмбеддинги)
- Вектор слова **фиксирован** после обучения: «забил» всегда один вектор

В LLaMA:
- Таблица эмбеддингов — это лишь **стартовая точка**
- Через 22 слоя attention каждый вектор **обновляется с учётом контекста**
- Вектор «забил» в начале → один вектор для всех значений
- Вектор «забил» после attention → разный в «нападающий забил гол» vs «плотник забил гвоздь»

Word2Vec дал статичные эмбеддинги. Трансформер даёт **динамичные, контекстно-зависимые**.

## 4. Токенизация: текст → числа

Нейросеть работает исключительно с числами. Первый шаг — превратить текст в последовательность целых чисел.

---

### 4.1 Почему не просто буквы?

Самая очевидная идея: назначить каждой букве один номер.

```
'а' → 0,  'б' → 1,  'в' → 2, ... 'я' → 32,  'a' → 33, ...
```

**Проблема:** слово «кошка» превращается в 5 токенов: `[к, о, ш, к, а]`.  
Фраза из 10 слов — это уже ~50 токенов. Длинный текст — тысячи токенов.

Каждый токен — это один «шаг» при обработке. Длиннее последовательность → дороже attention (O(n²) по времени и памяти). Буквенный токенизатор делает вход в 4–5 раз длиннее, чем нужно.

Кроме того, буква «к» сама по себе не несёт почти никакого смысла. Модель должна «собирать» слово из букв вместо того, чтобы сразу работать с целыми смысловыми единицами.

---

### 4.2 Почему не просто целые слова?

Другая крайность: один токен = одно слово.

**Проблема 1: огромный словарь.**  
В русском языке существуют сотни тысяч словоформ. Даже если взять только частые — словарь уже 500 000+ слов. Это значит таблица эмбеддингов размером `(500 000, 2048)` ≈ 4 ГБ — только для одной таблицы.

**Проблема 2: неизвестные слова (OOV).**  
В тестовых данных всегда встретится слово, которого не было в обучающем корпусе. Например, новое имя собственное, техницизм или опечатка. Слово-словарь не умеет с этим справиться.

**Проблема 3: морфология.**  
«кошка», «кошку», «кошке», «кошками» — четыре разных токена без какой-либо связи между ними в словаре.

---

### 4.3 Идея: субсловный токенизатор

Нужен компромисс:
- Частые слова → один токен («кошка» → `кошка`)
- Редкие слова → несколько токенов из известных кусков («трансформер» → `транс` + `форм` + `ер`)
- Неизвестные слова → всегда можно разбить до отдельных символов

Это называется **subword tokenization**. Самый популярный алгоритм — **BPE (Byte Pair Encoding)**.

---

### 4.4 Алгоритм BPE: пошагово

Обучение BPE начинается с вычисления множества уникальных слов в корпусе, после чего строится базовый словарь из всех символов, используемых для записи этих слов. Возьмём простой пример — корпус из пяти слов с частотами:

```
("hug", 10), ("pug", 5), ("pun", 12), ("bun", 4), ("hugs", 5)
```

Это означает: слово `"hug"` встречается 10 раз, `"pug"` — 5 раз, `"pun"` — 12 раз, и т.д.

**Начальный словарь** — набор всех отдельных символов:

```
Словарь: ["b", "g", "h", "n", "p", "s", "u"]
```

Разбиваем каждое слово на символы:

```
("h" "u" "g", 10), ("p" "u" "g", 5), ("p" "u" "n", 12), ("b" "u" "n", 4), ("h" "u" "g" "s", 5)
```

---

**Шаг 1: находим самую частую пару**

Подсчитываем частоты всех соседних пар:

| Пара | Встречается в | Итого |
|------|--------------|-------|
| `("h", "u")` | "hug" (×10) + "hugs" (×5) | **15** |
| `("u", "g")` | "hug" (×10) + "pug" (×5) + "hugs" (×5) | **20** ← МАКСИМУМ |
| `("p", "u")` | "pug" (×5) + "pun" (×12) | 17 |
| `("u", "n")` | "pun" (×12) + "bun" (×4) | 16 |
| `("b", "u")` | "bun" (×4) | 4 |
| `("g", "s")` | "hugs" (×5) | 5 |

Самая частая пара: `("u", "g")` — 20 раз.  
Создаём правило слияния: `("u", "g") → "ug"` и добавляем `"ug"` в словарь.

```
Словарь: ["b", "g", "h", "n", "p", "s", "u", "ug"]
Корпус:  ("h" "ug", 10), ("p" "ug", 5), ("p" "u" "n", 12), ("b" "u" "n", 4), ("h" "ug" "s", 5)
```

---

**Шаг 2:**

Теперь самая частая пара — `("u", "n")`: встречается в "pun" (×12) + "bun" (×4) = **16** раз.  
Правило: `("u", "n") → "un"`.

```
Словарь: ["b", "g", "h", "n", "p", "s", "u", "ug", "un"]
Корпус:  ("h" "ug", 10), ("p" "ug", 5), ("p" "un", 12), ("b" "un", 4), ("h" "ug" "s", 5)
```

---

**Шаг 3:**

Теперь самая частая пара — `("h", "ug")`: встречается в "hug" (×10) + "hugs" (×5) = **15** раз.  
Правило: `("h", "ug") → "hug"` — первый трёхбуквенный токен!

```
Словарь: ["b", "g", "h", "n", "p", "s", "u", "ug", "un", "hug"]
Корпус:  ("hug", 10), ("p" "ug", 5), ("p" "un", 12), ("b" "un", 4), ("hug" "s", 5)
```

---

Продолжаем так до тех пор, пока словарь не достигнет нужного размера. Итоговый список правил слияния для этого мини-корпуса:

```
правило 1:  ("u", "g")   → "ug"
правило 2:  ("u", "n")   → "un"
правило 3:  ("h", "ug")  → "hug"
правило 4:  ("hug", "s") → "hugs"
правило 5:  ("p", "ug")  → "pug"
...
```

> **✏️ Попробуйте сами!** Какое слияние будет следующим на шаге 4?

---

### 4.5 Токенизация нового текста

Токенизация следует тому же процессу, что и обучение: к новому тексту применяются выученные правила **строго в том же порядке**.

```
Шаги:
1. Нормализация текста
2. Предварительное разбиение на слова (pre-tokenization)
3. Разбивка каждого слова на отдельные символы
4. Последовательное применение правил слияния
```

**Пример** — токенизируем слово `"bug"` с тремя выученными правилами:

```
правило 1: ("u", "g") → "ug"
правило 2: ("u", "n") → "un"
правило 3: ("h", "ug") → "hug"
```

```
"bug"  →  ["b", "u", "g"]       (разбиваем на символы)
         правило 1 ("u","g")→"ug":  ["b", "ug"]   ✓
         правило 2 ("u","n")       :  нет пары, пропуск
         правило 3 ("h","ug")      :  нет "h" рядом с "ug", пропуск
         
Итог: ["b", "ug"]
```

А вот `"mug"` токенизируется как `["[UNK]", "ug"]`, поскольку буква `"m"` отсутствовала в базовом словаре.

> **⚠️ Ключевое свойство:** порядок правил критически важен! Правило 3 (`"h"` + `"ug"` → `"hug"`) зависит от результата правила 1 — если бы `"ug"` не было создано на шаге 1, правило 3 никогда бы не сработало.

---

### 4.6 Python-реализация BPE

Реализуем алгоритм с нуля. Это не оптимизированная версия — цель показать, как работает алгоритм изнутри.

**Шаг 1: подготовка корпуса**

```python
from collections import defaultdict

# Корпус: слово → частота
word_freqs = {
    "hug": 10,
    "pug": 5,
    "pun": 12,
    "bun": 4,
    "hugs": 5,
}

# Разбиваем каждое слово на символы
splits = {word: list(word) for word in word_freqs}
print(splits)
# {'hug': ['h', 'u', 'g'], 'pug': ['p', 'u', 'g'], ...}
```

**Шаг 2: функция подсчёта частот пар**

```python
def compute_pair_freqs(splits, word_freqs):
    pair_freqs = defaultdict(int)
    for word, freq in word_freqs.items():
        split = splits[word]
        for i in range(len(split) - 1):
            pair = (split[i], split[i + 1])
            pair_freqs[pair] += freq
    return pair_freqs

pair_freqs = compute_pair_freqs(splits, word_freqs)
# ('u', 'g'): 20  ← самая частая
# ('h', 'u'): 15
# ('u', 'n'): 16
# ...
```

**Шаг 3: функция слияния пары**

```python
def merge_pair(a, b, splits):
    for word in splits:
        split = splits[word]
        new_split = []
        i = 0
        while i < len(split):
            if i < len(split) - 1 and split[i] == a and split[i+1] == b:
                new_split.append(a + b)
                i += 2
            else:
                new_split.append(split[i])
                i += 1
        splits[word] = new_split
    return splits
```

**Шаг 4: основной цикл обучения**

```python
vocab_size = 15   # желаемый размер словаря
vocab = list(set(c for word in word_freqs for c in word))  # начальный алфавит
merges = {}       # словарь правил слияния

while len(vocab) < vocab_size:
    pair_freqs = compute_pair_freqs(splits, word_freqs)
    
    # Находим самую частую пару
    best_pair = max(pair_freqs, key=pair_freqs.get)
    
    # Сливаем пару во всём корпусе
    splits = merge_pair(*best_pair, splits)
    
    # Запоминаем правило и обновляем словарь
    new_token = best_pair[0] + best_pair[1]
    merges[best_pair] = new_token
    vocab.append(new_token)

print(merges)
# {('u', 'g'): 'ug', ('u', 'n'): 'un', ('h', 'ug'): 'hug', ...}
```

**Шаг 5: токенизация нового текста**

```python
def tokenize(text, merges):
    # Разбиваем текст на слова, каждое — на символы
    words = text.split()
    splits = [list(word) for word in words]
    
    # Применяем правила слияния в том же порядке, в котором они учились
    for pair, merged in merges.items():
        for idx, split in enumerate(splits):
            i = 0
            while i < len(split) - 1:
                if split[i] == pair[0] and split[i+1] == pair[1]:
                    split = split[:i] + [merged] + split[i+2:]
                else:
                    i += 1
            splits[idx] = split
    
    return sum(splits, [])

print(tokenize("hug pun bug", merges))
# ['hug', 'p', 'un', 'b', 'ug']
#   ↑ целый токен   ↑ "b" остался — не было в обучающих словах
```

---

### 4.7 Почему именно такие токены?

Вот ключевой вывод:

> **Токен появляется = потому что его составляющие часто стоят рядом в корпусе.**

- `"ug"` появилось потому что `"u"` + `"g"` встречались рядом **20 раз** (самая частая пара)
- `"hug"` стало токеном потому что `"h"` + `"ug"` встречается в словах "hug" и "hugs" суммарно 15 раз
- `"pun"` разбивается как `["p", "un"]`, а не один токен — потому что пара `("p", "un")` встречается реже, чем другие

Если бы слово `"hug"` встречалось всего 2 раза, пара `("h", "ug")` имела бы низкую частоту и `"hug"` никогда бы не стало токеном. **Токены — прямое отражение частотных паттернов языка.**

**Практический пример для русского:**
- «привет» — очень частое слово → с большой вероятностью один токен
- «трансформер» — редкое → разбивается: `транс` + `форм` + `ер`
- «ChatGPT» — не в обучающем корпусе → `Chat` + `G` + `PT` или даже по буквам
- числа: `«1234»` → `[1][2][3][4]` — каждая цифра отдельно, отсюда плохое арифметическое мышление LLM

---



### 4.8 Размер словаря: компромисс

Сколько слияний делать — гиперпараметр. Для TinyLlama: **32 000**.

| vocab_size | Плюсы | Минусы |
|------------|-------|--------|
| 1 000 (мало) | Простота | Длинные последовательности, много символов |
| 32 000 (средне) | Баланс для большинства языков | — |
| 100 000 (много) | Короткие последовательности | Большая таблица эмбеддингов, редкие токены |

GPT-4 использует **100 256** токенов. Llama-3 — **128 256**.

## 5. LLaMA целиком: математическая запись

Прежде чем разбирать детали — один раз запишем модель **целиком** в математической форме.  
Это можно нарисовать на доске одной схемой и потом «раскрывать» каждый блок отдельно.

---

### 5.1 Обозначения (размерности TinyLlama)

| Символ | Смысл | TinyLlama |
|--------|-------|-----------|
| $V$ | размер словаря | 32 000 |
| $d$ | hidden size | 2 048 |
| $L$ | число слоёв | 22 |
| $H$ | число Q-голов | 32 |
| $H_{KV}$ | число KV-голов | 4 |
| $d_h$ | размер одной головы, $d/H$ | 64 |
| $n$ | длина последовательности | ≤ 2 048 |
| $d_{ff}$ | размер FFN (промежуточный), ≈ $8d/3$, выровненный | 5 632 |

---

### 5.2 Схема модели (одна «картинка для доски»)

```
Вход: токены  [t₁, t₂, ..., tₙ]   ∈ ℤⁿ            (целые числа, n ≤ 2048)
        │
        ▼
┌───────────────┐
│  Embedding    │   X = E[t]   ∈ ℝ^{n × 2048}
└───────┬───────┘
        │
        ▼   повторить 22 раза
┌──────────────────────────────────────────┐
│  Transformer Block ℓ                     │
│                                          │
│  h = X + Attention( RMSNorm(X) )        │
│  X = h + FFN( RMSNorm(h) )              │
└──────────────────────────────────────────┘
        │
        ▼
┌───────────────┐
│  RMSNorm      │   X_final = RMSNorm(X)   ∈ ℝ^{n × 2048}
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  lm_head      │   logits = X_final · Eᵀ  ∈ ℝ^{n × 32000}
└───────┬───────┘
        │
        ▼
   softmax → p ∈ ℝ^{n × 32000}    (вероятности следующего токена)
```

> В LLaMA веса `lm_head` **совпадают** с матрицей эмбеддингов `E` — это называется **weight tying**.

---

### 5.3 Блок за блоком: формулы

#### Embedding (lookup)

$$X = E[\,t\,] \quad \in \mathbb{R}^{n \times 2048}$$

$E \in \mathbb{R}^{32000 \times 2048}$ — обучаемая таблица. Операция — просто выбрать строки.

---

#### RMSNorm

Вместо LayerNorm используется **Root Mean Square Norm** — нет вычитания среднего, только масштабирование:

$$\text{RMSNorm}(x) = \frac{x}{\text{RMS}(x)} \cdot \gamma, \quad \text{RMS}(x) = \sqrt{\frac{1}{2048}\sum_{i=1}^{2048} x_i^2 + \varepsilon}$$

$\gamma \in \mathbb{R}^{2048}$ — обучаемый вектор масштаба, $\varepsilon$ — маленькая константа для стабильности.

---

#### Позиционные эмбеддинги

Трансформер обрабатывает все токены одновременно и сам по себе не знает, кто стоит первым, а кто последним. Чтобы передать позицию, к вектору каждого токена **прибавляют позиционный вектор**:

$$X_m = E[t_m] + P_m$$

где $P_m \in \mathbb{R}^{2048}$ — позиционный вектор для позиции $m$. Он не обучается (в классическом варианте) и вычисляется по формуле:

$$P_{m,\, 2k} = \sin\!\left(\frac{m}{10000^{2k/2048}}\right), \quad P_{m,\, 2k+1} = \cos\!\left(\frac{m}{10000^{2k/2048}}\right)$$

Чётные измерения — синусы, нечётные — косинусы, частота убывает с ростом $k$. После прибавления вектор токена «знает» на какой он позиции.

> **В LLaMA вместо этого используется RoPE** (Rotary Positional Embedding) — более хитрый способ: позиция не прибавляется к эмбеддингу, а вкручивается в Q и K вращением пар координат уже внутри attention. Ровно тот же смысл — дать модели знать о позиции, но через другой механизм. RoPE заслуживает отдельного разбора.

---

#### Grouped Query Attention (GQA)

В LLaMA Q-проекций 32 (одна на каждую голову), а KV-проекций всего 4. Каждые $8$ Q-голов делят одну пару K, V.

Проекции применяются к $X_{\text{norm}} = \text{RMSNorm}(X) \in \mathbb{R}^{n \times 2048}$:

$$Q = X_{\text{norm}} \cdot W_Q \in \mathbb{R}^{n \times 2048}, \quad W_Q \in \mathbb{R}^{2048 \times 2048}$$

$$K = X_{\text{norm}} \cdot W_K \in \mathbb{R}^{n \times 256}, \quad W_K \in \mathbb{R}^{2048 \times 256} \quad (4 \text{ головы} \times 64)$$

$$V = X_{\text{norm}} \cdot W_V \in \mathbb{R}^{n \times 256}, \quad W_V \in \mathbb{R}^{2048 \times 256}$$

Разбиваем Q на 32 головы по 64, K и V — на 4 головы по 64, затем каждую KV-голову копируем на 8 Q-голов:

```
Q  → (32, n, 64)
K' = repeat(K, 8)  → (32, n, 64)     # 4 KV-головы → 32 копии
V' = repeat(V, 8)  → (32, n, 64)
```

Для каждой из 32 голов:

$$A^{(h)} = \text{softmax}\!\left(\frac{Q'^{(h)} \cdot K'^{(h)\top}}{\sqrt{64}} + M\right) \in \mathbb{R}^{n \times n}$$

где $M$ — causal mask ($M_{ij} = 0$ если $j \le i$, иначе $-\infty$).

$$O^{(h)} = A^{(h)} \cdot V'^{(h)} \in \mathbb{R}^{n \times 64}$$

Конкатенируем все 32 головы и применяем выходную проекцию:

$$\text{Attention}(X_{\text{norm}}) = \text{concat}(O^{(1)}, \ldots, O^{(32)}) \cdot W_O \in \mathbb{R}^{n \times 2048}$$

где $W_O \in \mathbb{R}^{2048 \times 2048}$.

**Single-head vs Multi-head: в чём разница?**

Ты правильно понял. Вот как это выглядит в сравнении:

```
── Single-head Attention (d = 2048) ──────────────────────────────────

X_norm: (n, 2048)
    │
    ├─ ·W_Q → Q: (n, 2048)
    ├─ ·W_K → K: (n, 2048)
    └─ ·W_V → V: (n, 2048)

scores = Q · Kᵀ / √2048       (n, n)
A      = softmax(scores + M)   (n, n)
O      = A · V                 (n, 2048)   ← готово, W_O не нужен

── Multi-head Attention (H=32 голов, d_h=64) ─────────────────────────

X_norm: (n, 2048)
    │
    ├─ ·W_Q → Q: (n, 2048) → разбиваем → 32 штуки Q⁽ʰ⁾: (n, 64)
    ├─ ·W_K → K: (n, 2048) → разбиваем → 32 штуки K⁽ʰ⁾: (n, 64)
    └─ ·W_V → V: (n, 2048) → разбиваем → 32 штуки V⁽ʰ⁾: (n, 64)

Для каждой головы h = 1..32:
    scores⁽ʰ⁾ = Q⁽ʰ⁾ · K⁽ʰ⁾ᵀ / √64   (n, n)
    A⁽ʰ⁾     = softmax(scores⁽ʰ⁾ + M)  (n, n)
    O⁽ʰ⁾     = A⁽ʰ⁾ · V⁽ʰ⁾             (n, 64)

concat(O⁽¹⁾, ..., O⁽³²⁾)               (n, 32×64) = (n, 2048)
    └─ ·W_O                             (n, 2048)   ← финальный выход
```

**Зачем вообще делить на головы?**

Каждая голова смотрит на «свой срез» пространства признаков и учится замечать свой тип зависимостей. Одна голова может специализироваться на синтаксических связях (субъект–глагол), другая — на ближних позиционных зависимостях, третья — на кореференции (местоимение → существительное). Если делать одну голову с $d=2048$, модель может это делать, но на практике несколько параллельных «маленьких attention» учатся лучше.

**$W_O$ — перемешивание информации между головами**

Это самая важная интуиция. Без $W_O$ каждая голова писала бы результат только в «свои» 64 измерения финального вектора — головы никогда бы не общались между собой:

```
Без W_O:
  После concat → (n, 2048)
  измерения 0..63    = то, что нашла голова 1   ┐
  измерения 64..127  = то, что нашла голова 2   │  полностью изолированы!
  измерения 128..191 = то, что нашла голова 3   ┘
  ...
```

$W_O \in \mathbb{R}^{2048 \times 2048}$ — это **полная** матрица. Умножение на неё означает, что каждое из 2048 выходных измерений является **линейной комбинацией всех 2048 входных** (то есть всех голов):

```
С W_O:
  выходное измерение 0 = 0.3·(голова 1, dim 7) + 0.1·(голова 5, dim 23) + 0.8·(голова 12, dim 4) + ...
  выходное измерение 1 = другая комбинация всех голов
  ...
```

**Конкретный пример:** голова 3 нашла, что токен «съел» синтаксически связан с «кот» (субъект). Голова 17 нашла, что «съел» семантически связан с «рыба» (объект). $W_O$ позволяет итоговому вектору токена «съел» **одновременно содержать оба знания** в скоординированном виде— она «знает», как правильно взвесить и скомбинировать открытия разных голов.

В метафоре: 32 головы — это 32 эксперта, каждый пишет своё заключение. $W_O$ — это редактор, который читает все 32 заключения и пишет один итоговый документ, переформулируя и совмещая информацию из всех источников.

---


#### SwiGLU FFN

Вместо стандартного $\text{ReLU}(x W_1) W_2$ используется **SwiGLU**:

$$\text{FFN}(x) = \bigl(\text{SiLU}(x W_{\text{gate}}) \odot x W_{\text{up}}\bigr) \cdot W_{\text{down}}$$

где:
- $W_{\text{gate}}, W_{\text{up}} \in \mathbb{R}^{2048 \times 5632}$ — две параллельные проекции «вверх»
- $W_{\text{down}} \in \mathbb{R}^{5632 \times 2048}$ — проекция обратно
- $\text{SiLU}(x) = x \cdot \sigma(x)$ — activation function
- $\odot$ — поэлементное умножение (gate управляет потоком)

---

#### Один Transformer Block целиком

$$h^\ell = X^{\ell-1} + \text{Attention}\!\bigl(\text{RMSNorm}(X^{\ell-1})\bigr)$$

$$X^\ell = h^\ell + \text{FFN}\!\bigl(\text{RMSNorm}(h^\ell)\bigr)$$

Это **Pre-Norm** архитектура: нормализация происходит **до** блока, а не после. Остаточная связь ($+$) обходит каждый блок.

---

#### Выходной слой

После 22 блоков:

$$X_{\text{final}} = \text{RMSNorm}(X^{22}) \in \mathbb{R}^{n \times 2048}$$

$$\text{logits} = X_{\text{final}} \cdot E^\top \in \mathbb{R}^{n \times 32000}$$

$$p_i = \text{softmax}(\text{logits}_i) \in \mathbb{R}^{32000} \quad \text{— вероятности следующего токена после позиции } i$$

---

### 5.4 Весь прямой проход одной формулой (псевдокод для доски)

```
X⁰ = E[t]                                         # Embedding: (n, 2048)

for ℓ = 1..22:
    h = X + Attn( RMSNorm(X) )                     # Attention + residual: (n, 2048)
    X = h  + FFN( RMSNorm(h) )                     # FFN + residual: (n, 2048)

где:
    Attn(z) = concat(softmax(Q'K'ᵀ/√64 + M)·V')·W_O
    Q' = RoPE( z·W_Q ),   K' = RoPE( z·W_K )
    FFN(z) = (SiLU(z·W_gate) ⊙ z·W_up) · W_down

logits = RMSNorm(X) · Eᵀ                          # lm_head: (n, 32000)
```

---

### 5.5 Что обучается (learnable параметры)

| Параметр | Размер | Назначение |
|----------|--------|-----------|
| $E$ | $(32000,\ 2048)$ | таблица эмбеддингов + lm\_head |
| $\gamma_{\text{attn}}, \gamma_{\text{ffn}}$ | $(2048,)$ × 44 шт | RMSNorm масштабы (2 на слой × 22 слоя) |
| $W_Q$ | $(2048,\ 2048)$ × 22 | Q-проекции |
| $W_K,\ W_V$ | $(2048,\ 256)$ × 22 | KV-проекции (4 головы × 64) |
| $W_O$ | $(2048,\ 2048)$ × 22 | выходная проекция |
| $W_{\text{gate}},\ W_{\text{up}}$ | $(2048,\ 5632)$ × 22 | FFN проекции вверх |
| $W_{\text{down}}$ | $(5632,\ 2048)$ × 22 | FFN проекция вниз |

Всего у TinyLlama-1.1B: **~1.1 миллиард** обучаемых параметров.

## 6. Эмбеддинги в LLaMA: таблица и lookup

В модели есть матрица **E** размером `(vocab_size × hidden_size)`:

```
E: (32000, 2048)
```

Каждой строке соответствует один токен. Когда мы «смотрим» токен с индексом `i`, мы просто берём `i`-ю строку этой матрицы. Это операция **lookup** — буквально «открыть таблицу на строке i».

```python
embed_tokens = nn.Embedding(32000, 2048)
hidden = embed_tokens(input_ids)   # (batch, seq_len) → (batch, seq_len, 2048)
```

**Важно:** в LLaMA таблица эмбеддингов обучается вместе со всей моделью. Начальные векторы случайны — они «дообучаются» через backpropagation так же, как веса attention и MLP.

### Числовые размерности TinyLlama

```
vocab_size    = 32 000   (размер словаря)
hidden_size   = 2 048    (размерность вектора токена)
seq_len       = до 2048  (максимальная длина контекста)
```

После эмбеддинга: тензор `(1, seq_len, 2048)` — каждому из seq_len токенов соответствует вектор из 2048 чисел.

## 6. Скалярное произведение: мера похожести

Это единственная математическая операция, которую нужно понять хорошо. Всё остальное строится на ней.

### Определение

Скалярное произведение двух векторов **a** и **b** одинаковой длины:

```
a · b = a₁·b₁ + a₂·b₂ + ... + aₙ·bₙ = Σᵢ aᵢ·bᵢ
```

Это просто: поэлементно умножить, потом сложить. Результат — одно число.

### Геометрический смысл

```
a · b = |a| · |b| · cos(θ)
```

где θ — угол между векторами.

| Угол θ | cos(θ) | Смысл |
|--------|--------|-------|
| 0° | 1 | Векторы одинаковы (максимальная похожесть) |
| 90° | 0 | Векторы перпендикулярны (не связаны) |
| 180° | -1 | Векторы противоположны |

**Практический вывод:** чем больше скалярное произведение — тем похожее векторы.

### Пример

```
«кошка» = [0.9, 0.1, 0.8]
«собака» = [0.8, 0.2, 0.7]
«стол»   = [0.1, 0.9, 0.2]

кошка · собака = 0.9·0.8 + 0.1·0.2 + 0.8·0.7 = 0.72 + 0.02 + 0.56 = 1.30  (похожи!)
кошка · стол   = 0.9·0.1 + 0.1·0.9 + 0.8·0.2 = 0.09 + 0.09 + 0.16 = 0.34  (не похожи)
```

### Матричное умножение

Если нужно вычислить скалярные произведения **всех пар** за один раз:

```
Q: матрица (seq_len × 64)    ← каждая строка = вектор токена
K: матрица (seq_len × 64)

Q · Kᵀ: матрица (seq_len × seq_len)
```

Элемент `[i, j]` результата = скалярное произведение векторов токенов `i` и `j`.

Это ключевая операция в attention-механизме.

## 7. Механизм внимания (Attention)

### Мотивация: проблема контекста

Рассмотрим два предложения:

```
1. «Он взял ключи и открыл дверь»
2. «Он взял ноты и открыл учебник»
```

Слово «открыл» в обоих случаях одинаковое, но его смысл разный — в первом случае открыл физически ключом, во втором — прочитал.

Если у каждого слова фиксированный вектор (только из таблицы эмбеддингов) — модель не различает эти контексты.

**Нужно:** вектор каждого слова должен обогатиться информацией о соседях.

### Q, K, V — три проекции

Для каждого токена из его вектора **x** вычисляются три новых вектора:

```
Q = x · Wq    (Query  — «что я ищу?»)
K = x · Wk    (Key    — «что я предлагаю?»)
V = x · Wv    (Value  — «что я отдаю, если нашли?»)
```

где **Wq**, **Wk**, **Wv** — обучаемые матрицы.

**Аналогия с поисковиком:**

Представьте YouTube:
- Вы вводите запрос → это **Query**
- У каждого видео есть название и теги → это **Key**
- Само содержимое видео → это **Value**

Attention делает то же самое, но для токенов в последовательности, и «одновременно» для всех токенов.

---

### Математика: проекции для всей последовательности

Пусть последовательность — матрица **X ∈ ℝ^{n×2048}**, где n = seq_len (до 2048).
Каждая строка Xᵢ ∈ ℝ^{2048} — вектор i-го токена.

Проекции применяются сразу ко всем токенам:

```
Q = X · Wq    ∈ ℝ^{n × 64}      Wq ∈ ℝ^{2048 × 64}     (одна голова)
K = X · Wk    ∈ ℝ^{n × 64}      Wk ∈ ℝ^{2048 × 64}
V = X · Wv    ∈ ℝ^{n × 64}      Wv ∈ ℝ^{2048 × 64}
```

**Матрица scores** (схожести всех пар токенов):

```
S = Q · Kᵀ / √64    ∈ ℝ^{n×n}

Sᵢⱼ = (Qᵢ · Kⱼ) / √64    — схожесть i-го запроса с j-м ключом
```

**Зачем √64?**

Предположим Qᵢ, Kⱼ ~ N(0, 1) (нормальные, независимые).  
Тогда: E[Qᵢ·Kⱼ] = 0, Var[Qᵢ·Kⱼ] = 64.  
Без деления стандартное отклонение = 8.  
Softmax от чисел с SD=8 → почти вырожденное распределение (один вес ≈ 1, остальные ≈ 0).  
После деления на √64 = 8: SD = 1, softmax нормальный.

**Шаг 3.** Softmax по строкам + causal mask M:

```
A = softmax(S + M)    ∈ ℝ^{n×n}

Aᵢⱼ = exp(Sᵢⱼ + Mᵢⱼ) / Σₖ exp(Sᵢₖ + Mᵢₖ)
```

где M — треугольная маска: Mᵢⱼ = 0 если j ≤ i, иначе -∞.

**Шаг 4.** Output:

```
O = A · V    ∈ ℝ^{n×64}

Oᵢ = Σⱼ Aᵢⱼ · Vⱼ    — взвешенная сумма, веса нормированы (сумма = 1)
```

### Формула целиком

```
Attention(Q, K, V) = softmax( Q · Kᵀ / √64 + M ) · V
```

Это формула из оригинальной статьи «Attention is All You Need» (Vaswani et al., 2017).

### Наглядный пример

Предложение: `«кот съел рыбу»`, seq_len = 3.

Вычислим attention для слова «съел» (строка 1, индексация с 0):

```
Q_съел · K_кот   = 8.2   ← кот важен: он субъект действия
Q_съел · K_съел  = 5.1   ← само слово умеренно важно
Q_съел · K_рыбу  = 7.8   ← рыба важна: она объект действия

После деления на √64 ≈ 0.125:
scores = [1.025, 0.637, 0.975]

После softmax:
weights = [0.38, 0.25, 0.37]    (сумма = 1)

output_съел = 0.38·V_кот + 0.25·V_съел + 0.37·V_рыбу
```

Результат: вектор «съел» теперь содержит информацию и о «коте», и о «рыбе»!

### Код (из `tiny_llama_clean.py`)

```python
def eager_attention_forward(query, key, value, attention_mask, scaling, num_key_value_groups):
    key_states = repeat_kv(key, num_key_value_groups)    # GQA expand
    value_states = repeat_kv(value, num_key_value_groups)
    
    attn_weights = torch.matmul(query, key_states.transpose(2, 3)) * scaling
    # scaling = head_dim**-0.5 = 1/√64
    
    if attention_mask is not None:
        attn_weights = attn_weights + attention_mask   # causal mask
    
    attn_weights = F.softmax(attn_weights, dim=-1, dtype=torch.float32).to(query.dtype)
    attn_output = torch.matmul(attn_weights, value_states)
    return attn_output
```

### Attention Maps: что видит модель

Матрица **A = softmax(QKᵀ/√64 + M)** — это не просто промежуточный результат вычислений. Её можно **визуализировать** как тепловую карту (heatmap): строки = токены-запросы, столбцы = токены-ключи, яркость = вес внимания.

**Для предложения «Кошка сидит на коврике» (causal, seq_len=5):**

```
Attention map (одна голова, пример):

         Кошка   сидит    на   коврике
Кошка  [  1.00    0.00   0.00    0.00 ]  ← видит только себя
сидит  [  0.71    0.29   0.00    0.00 ]  ← «сидит» смотрит на «Кошка»
на     [  0.10    0.55   0.35    0.00 ]  ← «на» смотрит на «сидит»
коврике[  0.35    0.05   0.12    0.48 ]  ← «коврике» смотрит на «Кошка» и себя

(Верхний треугольник = 0 из-за causal mask — нельзя смотреть вперёд)
```

**Что бывает в реальности (из исследований):**

| Паттерн головы | Что делает |
|----------------|-----------|
| **Positional head** | Смотрит на соседний токен: A[i, i-1] ≈ 1 |
| **Copy head** | Копирует информацию: A[i, j] ≈ 1 если token_i = token_j |
| **Induction head** | Предсказывает продолжение паттерна: если видел «AB», при «A» смотрит на все «B» |
| **Syntactic head** | Следит за субъектом/объектом через всё предложение |
| **BOS head** | Всегда смотрит на первый токен (BOS) — «мусорная корзина» для неважных токенов |

**Как извлечь attention maps из кода:**

```python
# В forward() модели можно вернуть attn_weights:
attn_weights = F.softmax(attn_weights, dim=-1)
# shape: (bsz, num_heads, seq_len_q, seq_len_k)
# → для визуализации берём [0, head_id, :, :] — матрица (seq, seq)
```

**Важная оговорка:**

Attention map ≠ «что думает модель». Большой вес A[i,j] означает: «токен i много брал из V[j]», но не «токен i семантически связан с j». Голова может иметь большие веса на позицию j просто потому что V[j] содержит нужную форму сигнала. Attention maps — инструмент интерпретации, но не объяснение.

## 8. Softmax: из чисел в вероятности

Softmax — функция, которая превращает вектор произвольных чисел в вектор вероятностей (неотрицательных, сумма = 1).

### Формула

Для вектора **x ∈ ℝⁿ**:

```
softmax(x)ᵢ = exp(xᵢ) / Σⱼ₌₁ⁿ exp(xⱼ)
```

В векторном виде: softmax(**x**) = exp(**x**) / (𝟏ᵀ exp(**x**)), где 𝟏 — вектор единиц.

### Пример

```
x = [2.0, 1.0, 0.1]

exp(x) = [e², e¹, e^0.1] ≈ [7.389, 2.718, 1.105]
Z = 7.389 + 2.718 + 1.105 = 11.212  (partition function)

softmax(x) = [7.389/11.212, 2.718/11.212, 1.105/11.212]
           = [0.659, 0.242, 0.099]   ← сумма = 1.0 ✓
```

### Свойства

1. **Монотонность:** xᵢ > xⱼ ⟹ softmax(x)ᵢ > softmax(x)ⱼ
2. **Ограниченность:** 0 < softmax(x)ᵢ < 1 для всех i
3. **Нормировка:** Σᵢ softmax(x)ᵢ = 1 (вероятностное распределение)
4. **Инвариантность к сдвигу:** softmax(x) = softmax(x + c) для любой константы c

### Численная стабильность (LogSumExp trick)

Прямое вычисление `exp(xᵢ)` при xᵢ = 1000 даёт overflow (числа ≫ 10^308 в float).

**Решение:** используем инвариантность к сдвигу:

```
M = max(x)            ← вычитаем максимум

softmax(x)ᵢ = exp(xᵢ - M) / Σⱼ exp(xⱼ - M)
```

При этом max(xᵢ - M) = 0, все exponents ≤ 1 → нет overflow.  
Знаменатель: log(Σⱼ exp(xⱼ - M)) + M — это **LogSumExp(x)**, численно стабильная форма.

**Пример overflow без трика:**
```
x = [1000, 999, 998]
exp(1000) = ∞                    ← float overflow!

С трюком: x - 1000 = [0, -1, -2]
exp([0, -1, -2]) = [1, 0.368, 0.135]    ← ок, числа ≤ 1
Z = 1.503
softmax = [0.665, 0.245, 0.090]    ✓
```

### Градиент softmax

Это важно понять для обучения. Производная softmax(x)ᵢ по xⱼ:

```
∂softmax(x)ᵢ / ∂xⱼ = softmax(x)ᵢ · (δᵢⱼ - softmax(x)ⱼ)
```

где δᵢⱼ = 1 если i=j, иначе 0 (символ Кронекера).  
В матричном виде: **Jacobian** = diag(p) - p·pᵀ, где p = softmax(x).

### В контексте attention

Каждая **строка** матрицы A = softmax(S + M) — самостоятельное вероятностное распределение:  
Aᵢ ∈ ℝⁿ, Aᵢ ≥ 0, Σⱼ Aᵢⱼ = 1.

Эта строка говорит: «токен i уделяет Aᵢⱼ·100% внимания токену j».

## 9. Causal Mask: не смотреть в будущее

### Проблема

Во время **обучения** мы даём модели весь текст сразу (это эффективнее). Но модель не должна «жульничать» — смотреть на future-токены при предсказании текущего.

### Решение: треугольная маска

Создаём матрицу `(seq_len × seq_len)` где верхний треугольник = `-∞`, нижний = `0`:

```
         поз_0  поз_1  поз_2  поз_3
поз_0  [   0    -∞     -∞     -∞  ]   ← «кот» видит только себя
поз_1  [   0     0     -∞     -∞  ]   ← «съел» видит кота и себя
поз_2  [   0     0      0     -∞  ]   ← «рыбу» видит трёх предыдущих
поз_3  [   0     0      0      0  ]   ← «.» видит всё предложение
```

Маску прибавляем к attention scores **до** softmax:

```
attn_weights = attn_weights + mask
```

Там где -∞: `exp(-∞) = 0` → после softmax эти позиции получают вес ровно 0.

**Результат:** каждый токен обращает внимание только на токены слева от себя (включая себя). Это называется **causal** (причинная) или **autoregressive** маска.

### Код

```python
def make_causal_mask(seq_len, past_len, dtype, device):
    total_len = past_len + seq_len
    mask = torch.ones(seq_len, total_len, dtype=torch.bool, device=device)
    mask = torch.tril(mask, diagonal=past_len)   # нижний треугольник = True
    min_val = torch.finfo(dtype).min             # -inf
    float_mask = torch.zeros(seq_len, total_len, dtype=dtype, device=device)
    float_mask = float_mask.masked_fill(~mask, min_val)
    return float_mask[None, None, :, :]          # (1, 1, seq_len, total_len)
```

## 10. Multi-Head Attention

### Мотивация

Одна «голова» внимания задаёт один тип вопроса: «кто субъект?» или «где глагол?». Но слovo имеет много аспектов одновременно: грамматика, семантика, ссылки, и т.д.

**Решение:** запустить **несколько attention параллельно**, каждый со своими матрицами Wq, Wk, Wv.

### Как это работает

В TinyLlama: `num_heads = 32`, `head_dim = 64`.

```
hidden_size = num_heads × head_dim = 32 × 64 = 2048
```

1. Проецируем эмбеддинг в Q, K, V:
   ```
   Q = x · Wq    (2048 → 32×64 = 2048)
   K = x · Wk    (2048 → 4×64  = 256)   ← GQA: только 4 головы для K!
   V = x · Wv    (2048 → 4×64  = 256)   ← то же для V
   ```

2. Делим Q на 32 «голов», K/V на 4 «голов».

3. Каждая голова вычисляет свой Attention независимо.

4. Результаты всех голов конкатенируются: `32 × (seq_len, 64) → (seq_len, 2048)`.

5. Проецируем обратно: `Wo: (2048 → 2048)`.

### Что учатся делать головы?

Исследования показали, что разные головы специализируются:
- Одни следят за синтаксическими зависимостями (подлежащее-сказуемое)
- Другие — за кореференцией (местоимение и его референт)
- Третьи — за позиционными паттернами (следующее слово, предыдущее слово)

Это возникает **само**, без какого-либо явного задания.

### Размерности в коде

```python
# В LlamaAttention.__init__:
self.q_proj = nn.Linear(2048, 32*64)   # (2048 → 2048)
self.k_proj = nn.Linear(2048,  4*64)   # (2048 →  256)  ← GQA
self.v_proj = nn.Linear(2048,  4*64)   # (2048 →  256)  ← GQA
self.o_proj = nn.Linear(32*64, 2048)   # (2048 → 2048)

# В forward():
query_states = self.q_proj(hidden_states).view(bsz, seq_len, 32, 64).transpose(1, 2)
# (bsz, seq_len, 2048) → (bsz, 32, seq_len, 64)
```

## 11. Residual Connections

### Проблема глубоких сетей

Нейросеть из 22 слоёв означает, что градиент должен пройти через 22 слоя назад. При перемножении многих матриц градиент либо:
- **Затухает** (vanishing gradient): становится почти 0, ранние слои не учатся
- **Взрывается** (exploding gradient): становится бесконечно большим

### Решение: обходной путь

```
выход = x + F(x)
```

где F(x) — это один слой (Attention или MLP), а x — входное значение, которое «обходит» слой напрямую.

```
  x ──────────────────────────────→ +
  │                                 │
  └── [Normalization] → [F(x)] ─────┘
                                    ↓
                                  выход
```

### Математика: почему residual спасает градиент

Пусть сеть из L слоёв: yₗ = yₗ₋₁ + Fₗ(yₗ₋₁).

Тогда по chain rule:

```
∂L/∂y₀ = ∂L/∂yₗ · Πₗ₌₁ᴸ (∂yₗ/∂yₗ₋₁)
        = ∂L/∂yₗ · Πₗ (1 + ∂Fₗ/∂yₗ₋₁)
```

Каждый множитель: **(1 + ∂Fₗ/∂yₗ₋₁)**.

**Без residual:** множитель — `∂Fₗ/∂yₗ₋₁`, — может быть < 1 (затухание) или > 1 (взрыв).

**С residual:** множитель = `1 + что-то`. Даже если `∂Fₗ/∂yₗ₋₁ ≈ 0` (слой ничего не делает), произведение всё равно ≈ 1.  
Градиент не исчезает — есть «прямой путь» через цепочку единиц.

**Раскрытое произведение** (упрощённо для 3 слоёв):
```
(1 + ∂F₁)(1 + ∂F₂)(1 + ∂F₃)
= 1 + ∂F₁ + ∂F₂ + ∂F₃ + ∂F₁∂F₂ + ∂F₁∂F₃ + ∂F₂∂F₃ + ...
```

Градиент — сумма из 2^L слагаемых! Каждое — «путь» через подмножество слоёв. Хотя бы некоторые пути пройдут через малое число слоёв и сохранят сигнал.

### Pre-LN vs Post-LN

**Оригинальный трансформер (2017):** Post-LN
```
x'  = LayerNorm(x + Sublayer(x))
```
Нормализация ПОСЛЕ residual. Проблема: при инициализации `∂Sublayer/∂x` большой → сумма `x + Sublayer(x)` имеет большую норму → нестабильно.

**LLaMA (Pre-LN):**
```
x' = x + Sublayer(Norm(x))
```
Нормализуем **вход** Sublayer. Норма входа ~1 → выход Sublayer имеет предсказуемую норму → стабильно.

Mathematically, при Pre-LN начальный градиент имеет норму O(1/L) vs O(1/√L) при Post-LN — оба малы, но Pre-LN стабильнее при больших L.

```python
# В LlamaDecoderLayer.forward():
residual = hidden_states
hidden_states = self.input_layernorm(hidden_states)    # Pre-LN: normalize BEFORE attention
hidden_states = self.self_attn(hidden_states, ...)
hidden_states = residual + hidden_states               # residual connection
```

## 12. Нормализация: RMSNorm и LayerNorm

### Зачем нормализация?

После каждого слоя значения могут расти или уменьшаться непредсказуемо. Через 22 слоя без нормализации числа могут стать `10^20` или `10^-20`. Нормализация «сбрасывает масштаб».

### LayerNorm (классический)

Для вектора **x ∈ ℝ^{2048}**:

```
μ = (1/d) Σᵢ xᵢ                              (среднее)
σ² = (1/d) Σᵢ (xᵢ - μ)²                      (дисперсия)

LN(x)ᵢ = γᵢ · (xᵢ - μ) / √(σ² + ε) + βᵢ
```

где **γ, β ∈ ℝ^{2048}** — обучаемые параметры (scale и shift), ε ≈ 1e-5.

**Обучаемых параметров:** 2d (γ и β).  
Для TinyLlama: 2 × 2048 = 4 096 параметров на один RMSNorm.

### RMSNorm (используется в LLaMA)

Убираем вычитание среднего (μ = 0 не вычитаем):

```
RMS(x) = √( (1/d) Σᵢ xᵢ² + ε )    (root mean square)

RMSNorm(x)ᵢ = γᵢ · xᵢ / RMS(x)
```

только **γ ∈ ℝ^{2048}** (нет β). Обучаемых параметров: 2048 на один слой.

**Почему работает без μ?**

После re-centering (вычитания μ) средние активации действительно ≈ 0, так что:  
hidden_states - μ ≈ hidden_states при μ ≈ 0.  
Empirically: убирание re-centering не ухудшает качество, но ускоряет на 10–15%.

### Числовой пример

```
x = [3.0, -1.0, 0.0, 2.0]    (d = 4)

# LayerNorm:
μ  = (3 - 1 + 0 + 2) / 4 = 1.0
σ² = ((3-1)² + (-1-1)² + (0-1)² + (2-1)²) / 4 = (4+4+1+1)/4 = 2.5
σ  = √2.5 ≈ 1.581
LN(x) = [(3-1)/1.581, (-1-1)/1.581, (0-1)/1.581, (2-1)/1.581] · γ + β
       ≈ [1.265, -1.265, -0.632, 0.632] · γ + β

# RMSNorm:
RMS = √((9+1+0+4)/4) = √3.5 ≈ 1.871
RMSNorm(x) = [3/1.871, -1/1.871, 0/1.871, 2/1.871] · γ
           ≈ [1.604, -0.535, 0, 1.069] · γ
```

```python
class LlamaRMSNorm(nn.Module):
    def forward(self, hidden_states):
        # hidden_states: (bsz, seq_len, hidden_size)
        hidden_states = hidden_states.to(torch.float32)          # upcast для точности
        variance = hidden_states.pow(2).mean(-1, keepdim=True)   # = RMS²  (без ε)
        hidden_states = hidden_states * torch.rsqrt(variance + self.variance_epsilon)
        return self.weight * hidden_states.to(input_dtype)       # scale γ
```

## 13. MLP-блок: SwiGLU

После attention каждый токен обрабатывается нейросетью **независимо** (без взаимодействия с соседями). Если attention «решает, на кого смотреть», то MLP «обрабатывает собранную информацию».

### Обычный FFN (для сравнения)

Для **x ∈ ℝ^{2048}**, W₁ ∈ ℝ^{5632×2048}, W₂ ∈ ℝ^{2048×5632}:

```
FFN(x) = W₂ · ReLU(W₁ · x + b₁) + b₂

ReLU(z) = max(0, z)    — нелинейность
```

Расширяем d → m (обычно m = 4d), применяем нелинейность, проецируем обратно m → d.

### Активационные функции: сравнение

| z | ReLU(z) | SiLU(z) | Derivative ReLU | Derivative SiLU |
|---|---------|---------|-----------------|------------------|
| -2 | 0 | -0.24 | 0 | -0.09 |
| -1 | 0 | -0.27 | 0 | 0.07 |
| 0 | 0 | 0 | **undef** | 0.5 |
| 1 | 1 | 0.73 | 1 | 1.09 |
| 2 | 2 | 1.76 | 1 | 1.09 |

**ReLU:** производная = 0 при z < 0 → нейрон «мёртв» если всегда получает отрицательный вход.  
**SiLU:** производная ненулевая везде, плавный переход через 0.

### SwiGLU (LLaMA)

Для **x ∈ ℝ^{2048}**, Wg, Wu ∈ ℝ^{5632×2048}, Wd ∈ ℝ^{2048×5632}:

```
gate(x) = SiLU(Wg · x)       ∈ ℝ^m     — «ворота», что пропустить
val(x)  = Wu  · x            ∈ ℝ^m     — «значения», что несём

SwiGLU(x) = Wd · (gate(x) ⊙ val(x))   ∈ ℝ^{2048}
```

где ⊙ — поэлементное умножение (Hadamard product).

**SiLU (Sigmoid Linear Unit):** σ(z) = 1/(1+e^{-z})
```
SiLU(z) = z · σ(z) = z / (1 + e^{-z})

d/dz SiLU(z) = σ(z) + z · σ(z)(1 - σ(z))
             = σ(z) · (1 + z(1 - σ(z)))
```

**Почему мультипликация gate и val?**

Идея из GLU (Gated Linear Units, Dauphin et al. 2017):
```
GLU(x) = (W₁·x) ⊙ σ(W₂·x)
```
ворота динамически «выключают» ненужные нейроны.  
SwiGLU заменяет σ (сигмоиду) на SiLU — более плавная нелинейность.

**Почему intermediate_size = 5632 (а не 2048×4 = 8192)?**

SwiGLU вводит третью матрицу: W_gate, W_up, W_down. 3 матрицы вместо 2.  
Чтобы суммарное число параметров (≈FLOPs) совпадало с обычным 2-слойным FFN:

```
Обычный FFN:   2 × d × 4d = 8d²      (W₁ и W₂)
SwiGLU FFN:    3 × d × m             (Wg, Wu, Wd)

Чтобы 3dm = 8d²:  m = 8d/3 ≈ 2.667d

d = 2048:  m = 2048 × 8/3 ≈ 5461.3 → округляем до 5632 (кратно 128)
```

```python
class LlamaMLP(nn.Module):
    def forward(self, x):
        # x:         (bsz, seq_len, 2048)
        gate = F.silu(self.gate_proj(x))   # SiLU(Wg·x)   (bsz, seq_len, 5632)
        val  = self.up_proj(x)             # Wu·x          (bsz, seq_len, 5632)
        return self.down_proj(gate * val)  # Wd·(gate⊙val) (bsz, seq_len, 2048)
```

## 14. Позиционное кодирование: RoPE

### Проблема

Attention вычисляет скалярные произведения между токенами — операция симметричная и не зависящая от порядка. Без дополнительной информации:

```
«кошка съела мышь» = «мышь съела кошку»   ← одинаковые attention scores!
```

Нужно добавить информацию о **позиции** каждого токена.

### Абсолютное позиционное кодирование (оригинальный трансформер)

Прибавляем к эмбеддингу фиксированный вектор PE(pos) ∈ ℝ^{2048}:

```
x_pos = x + PE(pos)

PE(pos, 2k)   = sin( pos / 10000^{2k/d} )
PE(pos, 2k+1) = cos( pos / 10000^{2k/d} )
```

k = 0..d/2-1 задаёт частоту. Низкие k → высокие частоты (быстро меняется с позицией), высокие k → медленно.

Проблема: вектор PE(pos) прибавляется статически. При длинах, не видевшихся при обучении — нет гарантий.

### RoPE (Rotary Positional Embedding)

Идея: **вращать** Q и K на угол пропорциональный позиции, не изменяя само embedding.

**Математика для одной комплексной пары (q₁, q₂) на позиции m:**

Представим q₁ + i·q₂ ∈ ℂ. Умножение на e^{iθₖm}:

```
(q₁ + iq₂) · e^{iθₖm} = (q₁ + iq₂)(cos θₖm + i sin θₖm)

Вещественная часть: q₁ cos θₖm - q₂ sin θₖm
Мнимая часть:       q₁ sin θₖm + q₂ cos θₖm
```

В матричном виде — это поворот вектора (q₁, q₂) на угол θₖm:

```
[q₁']   [cos θₖm   -sin θₖm] [q₁]
[q₂'] = [sin θₖm    cos θₖm] [q₂]
```

### Полная форма для d-мерного вектора

Для вектора q ∈ ℝ^{2048} на позиции m задаём вращающую матрицу **Rₘ ∈ ℝ^{2048×2048}** — блочно-диагональную:

```
Rₘ = diag(R(θ₀m), R(θ₁m), ..., R(θ_{d/2-1} m))

где R(θ) = [[cos θ, -sin θ],   (2×2 матрица вращения)
            [sin θ,  cos θ]]
```

Тогда повёрнутый вектор: **q'** = Rₘ **q**.

**Частоты θₖ:**

```
θₖ = 1 / (base^{2k/d}),     k = 0, 1, ..., d/2-1

base = 10000 (rope_theta в TinyLlama)

θ₀  = 1/10000⁰   = 1          (быстрая частота)
θ₁  = 1/10000^{2/64} ≈ 0.61
θ₃₁ = 1/10000^{1}  = 0.0001   (медленная частота)
```

### Ключевое свойство: относительность позиций

**Теорема:** после RoPE-вращения скалярное произведение Qₘ·Kₙ зависит только от (m−n):

```
Qₘ · Kₙ = (Rₘ q) · (Rₙ k)
         = qᵀ Rₘᵀ Rₙ k
         = qᵀ R_{n-m} k      (т.к. Rₘᵀ Rₙ = R_{n-m} для ортогональных R)
```

Таким образом, attention score зависит от разности позиций **(n − m)**, а не от абсолютных позиций. Это называется **relative positional encoding** и хорошо обобщается на большие контексты.

**Аналогия с часами:** секундная стрелка (k=0, θ=1) быстро вращается — кодирует соседей. Часовая (k=31, θ=0.0001) — медленно — кодирует дальних. Вместе d/2=32 «стрелок» однозначно кодируют позицию до 2048.

### Применение в коде

```python
# rotate_half реализует [-x₂, x₁] — сдвиг компонент как при умножении на i
def rotate_half(x):
    x1 = x[..., :x.shape[-1]//2]           # первые d/2 dim: Re часть
    x2 = x[..., x.shape[-1]//2:]           # вторые d/2 dim: Im часть
    return torch.cat((-x2, x1), dim=-1)    # [-Im, Re] ≡ умножение на i

# apply_rotary_pos_emb реализует q' = q·cos + rotate_half(q)·sin
#   = [q₁, q₂]·cos + [-q₂, q₁]·sin
#   = [q₁ cos - q₂ sin,  q₂ cos + q₁ sin]  ← это и есть поворот!
def apply_rotary_pos_emb(q, k, cos, sin):
    cos = cos.unsqueeze(1)    # (bsz, 1, seq, d) — broadcast по heads
    sin = sin.unsqueeze(1)
    q_embed = q * cos + rotate_half(q) * sin
    k_embed = k * cos + rotate_half(k) * sin
    return q_embed, k_embed
```

## 15. Один декодер-блок целиком

Собираем всё вместе. Один блок LlamaDecoderLayer:

```
Вход: x  (bsz, seq_len, 2048)
              │
              │   ┌─── сохраняем копию (residual) ───────────────┐
              │                                                   │
              ↓                                                   │
         RMSNorm(x)                                              │
              │                                                   │
              ↓                                                   │
    ┌─── Self-Attention ───┐                                      │
    │  1. Q = x · Wq       │                                      │
    │  2. K = x · Wk       │                                      │
    │  3. V = x · Wv       │                                      │
    │  4. RoPE(Q, K)       │                                      │
    │  5. scores = Q·Kᵀ/√d │                                      │
    │  6. + causal mask     │                                      │
    │  7. softmax(scores)  │                                      │
    │  8. output = w·V     │                                      │
    │  9. concat heads     │                                      │
    │  10. · Wo            │                                      │
    └──────────────────────┘                                      │
              │                                                   │
              ↓                                                   │
              + ← ─────────────────────────────────────────────── ┘
              │   (residual connection)
              │
              │   ┌─── сохраняем копию (residual) ───────────────┐
              │                                                   │
              ↓                                                   │
         RMSNorm(x')                                             │
              │                                                   │
              ↓                                                   │
    ┌────── MLP ────────┐                                         │
    │  gate = SiLU(W_g·x)│                                        │
    │  val  = W_up · x   │                                        │
    │  out  = W_d·(g·v)  │                                        │
    └────────────────────┘                                        │
              │                                                   │
              ↓                                                   │
              + ← ─────────────────────────────────────────────── ┘
              │
              ↓
         Выход (bsz, seq_len, 2048)   ← та же форма!
```

**Замечательный факт:** форма тензора не меняется! `(bsz, seq_len, 2048)` на входе → `(bsz, seq_len, 2048)` на выходе. Это позволяет просто нанизать 22 таких блока один за другим.

## 16. Полная архитектура LLaMA

### Общая схема

```
Текст (строка)
    │
    ↓ Токенизатор (BPE, 32K словарь)
    │
input_ids: (1, seq_len)         — целые числа [0..31999]
    │
    ↓ Embedding (таблица 32000 × 2048)
    │
hidden: (1, seq_len, 2048)
    │
    ↓ RoPE-вычисление (cos и sin таблицы)
    │
    ├── Decoder Layer 0
    ├── Decoder Layer 1
    ├── ...
    └── Decoder Layer 21     ← 22 блока, hidden_size = 2048 на протяжении всего
    │
    ↓ Final RMSNorm
    │
hidden: (1, seq_len, 2048)
    │
    ↓ lm_head: Linear(2048 → 32000, bias=False)
    │
logits: (1, seq_len, 32000)  — для каждого токена: оценка всех 32K следующих токенов
    │
    ↓ argmax (при greedy decoding)
    │
next_token: integer           — индекс наиболее вероятного следующего токена
```

### Параметры TinyLlama-1.1B

| Гиперпараметр | Значение | Смысл |
|---------------|----------|-------|
| `vocab_size` | 32 000 | Размер словаря |
| `hidden_size` | 2 048 | Размерность вектора токена |
| `num_hidden_layers` | 22 | Количество декодер-блоков |
| `num_attention_heads` | 32 | Голов внимания (Q) |
| `num_key_value_heads` | 4 | Голов внимания (K, V) — GQA |
| `head_dim` | 64 | Размерность каждой головы (2048/32) |
| `intermediate_size` | 5 632 | Ширина MLP-слоя |
| `max_position_embeddings` | 2 048 | Максимальная длина контекста |
| `rope_theta` | 10 000 | База для RoPE-частот |
| `rms_norm_eps` | 1e-5 | Epsilon для RMSNorm |

### Подсчёт параметров

```
Embedding:           32000 × 2048               = 65.5M

Один блок:
  q_proj:            2048 × 2048                = 4.19M
  k_proj:            2048 ×  256                = 0.52M
  v_proj:            2048 ×  256                = 0.52M
  o_proj:            2048 × 2048                = 4.19M
  gate_proj:         2048 × 5632                = 11.5M
  up_proj:           2048 × 5632                = 11.5M
  down_proj:         5632 × 2048                = 11.5M
  2 × rmsnorm:       2 × 2048                  = 0.004M
  Итого на блок:                                ≈ 44M

22 блока:                                       = 22 × 44M = 968M
Final norm:          2048                       = 0.002M
lm_head:             2048 × 32000              = 65.5M

Итого:               65.5 + 968 + 65.5         ≈ 1.1B ✓
```

## 17. KV Cache: почему генерация быстрая

### Проблема без кэша

При генерации каждый шаг: у нас уже сгенерировано N токенов, мы хотим предсказать (N+1)-й.

Без оптимизации: прогоняем через модель всю последовательность длиной N снова с нуля. Каждый шаг занимает время O(N²) — квадратично от длины контекста.

### Идея: кэшировать K и V

Заметим: K и V для токенов 0..N-1 не меняются от шага к шагу! (Их не трогает новый токен.) Можно сохранить их один раз и переиспользовать.

```
Шаг 1 (prefill):
  Подаём весь промпт (токены 0..31)
  Вычисляем K, V для всех 32 токенов → сохраняем в кэш
  Получаем следующий токен (токен 32)

Шаг 2 (decode):
  Подаём только токен 32
  Вычисляем Q, K, V только для него
  K[32], V[32] → добавляем в кэш
  Достаём из кэша K[0..32], V[0..32]
  Считаем attention(Q[32], K[0..32], V[0..32])
  Получаем токен 33

...каждый шаг O(N) вместо O(N²)!
```

### Структура кэша

```python
class DynamicKVCache:
    def __init__(self):
        self.key_cache: list[torch.Tensor] = []     # per layer
        self.value_cache: list[torch.Tensor] = []

    def update(self, key_states, value_states, layer_idx):
        # Конкатенируем новые K, V к уже накопленным
        self.key_cache[layer_idx] = torch.cat(
            [self.key_cache[layer_idx], key_states], dim=2  # по seq dim
        )
        return self.key_cache[layer_idx], self.value_cache[layer_idx]
```

### Размер кэша

Для одного слоя:

```
K: (bsz, num_kv_heads, seq_len, head_dim) = (1, 4, 2048, 64) = 524 288 чисел
V: то же самое

Для 22 слоёв:  22 × 2 × 524 288 × 2 байта (bf16) ≈ 44 MB
```

Это позволяет удерживать контекст 2048 токенов в ~44 MB видеопамяти.

### Измеренное ускорение (RTX 5090, TinyLlama)

```
С KV cache:    191.8 tok/s
Без KV cache:   56.5 tok/s
Ускорение:       3.4×
```

## 18. Grouped Query Attention (GQA)

### Стандартный MHA (Multi-Head Attention)

В оригинальном трансформере: `num_q_heads = num_kv_heads = 32`.

```
K cache на слой: (bsz, 32, seq_len, 64)  ← 32 головы K
V cache на слой: (bsz, 32, seq_len, 64)  ← 32 головы V
```

При большом батче или длинном контексте — KV cache огромный.

### GQA: меньше KV-голов

**Идея:** разделить 32 Q-головы на 4 группы по 8. Каждая группа использует **одну общую** K-голову и **одну общую** V-голову.

```
TinyLlama: num_q_heads = 32, num_kv_heads = 4
Группировка: каждая KV-голова делится 8 Q-головами (32/4 = 8)
```

**Экономия памяти KV cache:** в 8 раз меньше, чем при MHA.

При вычислении attention K/V «размножаются» через `repeat_kv`:

```python
def repeat_kv(hidden_states, n_rep):
    # (bsz, 4, seq, 64) → (bsz, 32, seq, 64)
    # каждую из 4 голов повторяем n_rep=8 раз
    hidden_states = hidden_states[:, :, None, :, :].expand(
        batch, num_kv_heads, n_rep, slen, head_dim
    )
    return hidden_states.reshape(batch, num_kv_heads * n_rep, slen, head_dim)
```

Это только **просмотр** (view) памяти — без копирования, бесплатно.

## 19. Обучение: как модель «учится»

### Loss функция: Cross-Entropy

Модель выдаёт `logits: (batch, seq_len, vocab_size)` — ненормализованные оценки для каждого токена.

Для одного токена на позиции t, если истинный токен имеет индекс yₜ:

```
logits_t  ∈ ℝ^V                      (V = vocab_size = 32 000)
P(w | x_{<t}) = softmax(logits_t)_w  (вероятность каждого слова)

Loss_t = -log P(yₜ | x_{<t})         (cross-entropy для позиции t)
```

Для всей последовательности длиной T — **среднее** по позициям:

```
L(θ) = -(1/T) Σₜ₌₁ᵀ log P_θ(yₜ | y₁, ..., yₜ₋₁)
```

### Авторегрессионная вероятность

Полная вероятность последовательности y = (y₁, y₂, ..., yₙ) по chain rule:

```
P(y₁, y₂, ..., yₙ) = P(y₁) · P(y₂|y₁) · P(y₃|y₁,y₂) · ... · P(yₙ|y₁..yₙ₋₁)
                    = Πₜ₌₁ⁿ P(yₜ | y_{<t})
```

Логарифм произведения — сумма логарифмов:

```
log P(y₁..yₙ) = Σₜ log P(yₜ | y_{<t})
```

Минимизировать negative log-likelihood = максимизировать вероятность последовательности.

### Perplexity: интерпретируемая метрика

Cross-entropy loss трудно интерпретировать напрямую (зависит от единиц). Perplexity:

```
PPL = exp(L) = exp( -(1/T) Σₜ log P(yₜ | y_{<t}) )
```

**Смысл:** «сколько слов» модель рассматривает как равновероятных при каждом предсказании.

| PPL | Смысл |
|-----|-------|
| 1 | Идеально — всегда угадывает правильно |
| 10 | Эффективно выбирает из ~10 слов |
| 100 | Из ~100 слов — случайный выбор среди частых |
| 32 000 | Равномерное распределение — ничему не научилась |

TinyLlama достигает PPL ≈ 5–8 на стандартных бенчмарках.

### Teacher Forcing

Во время обучения модель получает **истинные** предыдущие токены, а не свои собственные предсказания:

```
Математически:
  Вход:   x  = (y₀, y₁, y₂, ..., yₙ₋₁)   ← истинные токены (сдвинуто влево)
  Целевое: y = (y₁, y₂, y₃, ..., yₙ)     ← сдвинуто вправо

  Для позиции t: P(yₜ | y₀..yₜ₋₁) вычисляется с истинными y₀..yₜ₋₁
```

Это делает обучение стабильным: нет накопления ошибок предсказания.

**Альтернатива (Scheduled Sampling):** иногда подавать собственные предсказания модели, постепенно убирая teacher forcing. Но обычные LLM используют чистый teacher forcing.

### Градиентный спуск

**Алгоритм (1 шаг):**

```
1. Forward pass:   ŷ = f(x; θ)        — предсказания модели
2. Loss:           L = CE(ŷ, y)        — среднее по батчу
3. Backward pass:  ∇_θ L              — градиент по всем параметрам
4. Обновление:     θ ← θ - η · ∇_θ L  — η = learning rate
```

**TinyLlama:** обучался с learning rate η ≈ 4×10⁻⁴ на пике, с cosine decay.

### Adam Optimizer

На практике используется Adam (Adaptive Moment Estimation, Kingma & Ba 2015):

```
mₜ = β₁ · mₜ₋₁ + (1 - β₁) · ∇L    (первый момент — скользящее среднее градиента)
vₜ = β₂ · vₜ₋₁ + (1 - β₂) · ∇L²   (второй момент — скользящее среднее квадрата)

m̂ₜ = mₜ / (1 - β₁ᵗ)              (коррекция смещения)
v̂ₜ = vₜ / (1 - β₂ᵗ)

θₜ = θₜ₋₁ - η · m̂ₜ / (√v̂ₜ + ε)
```

Типичные гиперпараметры LLM: β₁ = 0.9, β₂ = 0.95, ε = 10⁻⁸.

**Зачем адаптивный lr?**  
Параметры с малыми градиентами (редкие токены в embedding) обновляются сильнее, параметры с большими — слабее. Это важно для обучения больших разреженных моделей.

### Gradient Clipping

Взрывающийся градиент: `‖∇L‖ > threshold`.

```
Если ‖∇L‖₂ > clip_norm:
    ∇L ← ∇L · (clip_norm / ‖∇L‖₂)

clip_norm = 1.0  (типично для LLM)
```

Это масштабирует **весь** вектор градиента так, чтобы его норма не превышала clip_norm, сохраняя направление.

### Итого: что происходит при обучении

```
Дано: корпус текста C = {x₁, x₂, ..., x|C|}
Параметры: θ ∈ ℝ^P (для TinyLlama: P ≈ 1.1 × 10⁹)

Цель: min_θ E_{x~C}[ -Σₜ log P_θ(xₜ | x_{<t}) ]

Итерации: ~3 × 10¹² токенов / batch_size  ≈ миллионы шагов
```

`1.1 миллиарда параметров × 3 триллиона токенов = 3.3 × 10²¹ операций умножения.`  
Это и есть «обучение» языковой модели.

### Что значит «модель знает»?

Знания модели — это **числа в матрицах**. Факт «столица Франции — Париж» закодирован в весах MLP или паттернах attention. Мы не можем указать пальцем «вот тут знание о Париже», но при правильном промпте оно активируется — нужные нейроны срабатывают, нужные attention паттерны возбуждаются.
